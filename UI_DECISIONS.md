# UI Decisions

# Общая философия

Приложение проектируется в первую очередь для использования на настольных компьютерах.

Приоритеты:

1. Удобство использования
2. Простота
3. Последовательность
4. Визуальная привлекательность

Любые визуальные изменения не должны ухудшать удобство работы.

---

# Навигация

Основная навигация:

- Каталог категорий
- Каталог продуктов
- Каталог брендов

▼ Администрирование

- Бренды
- Категории
- OpenAPI

Раздел «Варианты» удалён из навигации.

Варианты управляются только внутри страниц продуктов.

---

# Рабочие страницы и администрирование

Рабочие страницы предназначены для повседневной работы пользователей.

Они должны содержать только действия, необходимые в обычном рабочем процессе.

Страницы администрирования содержат:

- создание;
- редактирование;
- удаление;
- настройки.

Административные действия не должны отображаться на рабочих страницах.

---

# Каталог брендов

Назначение:

Просмотр и выбор брендов.

Содержит:

- поиск;
- алфавитную навигацию;
- список брендов.

Не содержит:

- создание;
- редактирование;
- удаление;
- отображение ID.

Все административные операции выполняются только через:

**Администрирование → Бренды**

---

# Сортировка брендов

Порядок сортировки:

1. Символы
2. Цифры
3. Латиница
4. Кириллица

Это правило должно использоваться во всём приложении.

---

# Макет интерфейса

Подход — Desktop First.

Интерфейсы должны:

- эффективно использовать горизонтальное пространство;
- минимизировать вертикальную прокрутку;
- оставаться компактными;
- избегать чрезмерно крупных элементов управления.

Списки должны быть компактными и легко просматриваться.

---

# Будущие возможности

Не добавлять элементы интерфейса «на будущее».

Если доступен только один вариант действия или сортировки — не отображать элементы управления, создающие впечатление выбора.

Каждый элемент интерфейса должен приносить реальную пользу пользователю.

---

# Администрирование брендов

Страница администрирования брендов не должна постоянно отображать форму редактирования.

Правила:

- создание выполняется через компактную форму;
- поиск расположен рядом с формой создания;
- редактирование начинается после выбора строки;
- интерфейс редактирования появляется только после выбора бренда;
- действия в таблице представлены компактными кнопками-иконками;
- текстовые кнопки действий в плотных административных таблицах использовать не следует.

---

# Действия в административных таблицах

Для действий используются компактные кнопки с иконками.

Правила:

- редактирование — иконка карандаша;
- удаление — иконка корзины;
- все кнопки должны иметь `title` и `aria-label`;
- кнопки визуально выравниваются по центру;
- текстовые кнопки в компактных таблицах не используются.

---

# Общие UI-компоненты

Повторно используемые компоненты следует выделять только после того, как они стабилизировались хотя бы на одном реальном экране.

Текущие общие шаблоны:

- заголовок страницы со счётчиком;
- компактная панель поиска;
- административная таблица;
- кнопки действий с иконками;
- сортировка каталогов по имени.

Цель — обеспечить единообразие интерфейса без масштабных переработок фронтенда.

---

# Каталог категорий

Каталог категорий использует иерархическую модель навигации.

Назначение страницы:

- просмотр структуры категорий;
- навигация по иерархии;
- выбор категории.

Рабочая страница не должна содержать:

- создание;
- редактирование;
- удаление;
- отображение ID;
- отображение parent_id.

Все административные операции выполняются только через раздел администрирования.

---

# Отображение иерархических данных

Иерархические структуры (категории, группы, разделы и т.п.) должны отображаться так, чтобы пользователь без дополнительных действий понимал:

- уровень вложенности;
- родительские элементы;
- дочерние элементы;
- текущий выбранный элемент;
- полный путь до выбранного элемента.

### Предпочтительное представление

Для desktop-интерфейсов предпочтительным является классическое вертикальное дерево.

Дерево должно обеспечивать:

- хорошо различимые уровни вложенности;
- раскрытие и сворачивание узлов;
- явное выделение выбранного элемента;
- подсветку цепочки родителей;
- стабильную структуру без визуальных скачков при раскрытии дерева.

Использование горизонтальных каскадных меню (mega menu, fly-out menu и аналогичных решений) для больших иерархий не рекомендуется, поскольку они ухудшают читаемость структуры и усложняют навигацию.

---

# Поиск в иерархии

Поиск является вспомогательным инструментом навигации и не заменяет само дерево.

Правила:

- поиск расположен в верхней части страницы;
- результаты отображают полный путь до категории;
- выбор результата автоматически раскрывает дерево до нужного узла;
- найденная категория становится активной;
- пользователь не должен терять контекст после перехода из поиска.

---

# UX-принципы для дерева

Работа с деревом должна минимизировать потерю контекста.

Раскрытие узлов не должно:

- ломать макет страницы;
- смещать основную рабочую область;
- затруднять понимание текущего положения пользователя.

Основные действия (просмотр, навигация, выбор категории) должны оставаться одинаково удобными независимо от глубины вложенности дерева.

---

# Каскадная навигация категорий

Каталог категорий использует каскадную навигацию по уровням вложенности.

Это решение допустимо, потому что глубина дерева ограничена:

- обычно 3 уровня;
- максимум 4 уровня.

Каскадная навигация применяется для быстрого перехода по категориям и последующего открытия списка продуктов конечной категории.

## Поведение

Правила:

- категории верхнего уровня отображаются в первой колонке;
- при выборе категории с дочерними элементами следующий уровень открывается справа;
- каждый следующий уровень открывается в соседней колонке;
- выбранная цепочка категорий должна быть визуально понятна;
- конечная выбранная категория должна быть выделена сильнее остальных;
- категории с дочерними элементами показывают chevron справа;
- breadcrumb отображает полный путь до выбранной категории.

## Визуальный стиль

Каскадный каталог не должен выглядеть как таблица.

Не использовать жесткие заметные вертикальные разделители между колонками.

Разделение уровней должно достигаться за счет:

- расстояния между колонками;
- аккуратного выравнивания;
- мягкого фонового контраста;
- hover- и active-состояний;
- визуальной группировки элементов.

Интерфейс должен оставаться легким и чистым.

Колонки не должны конкурировать с содержимым страницы. Основной акцент должен быть на выбранном пути и конечной категории.

## Поиск

Поиск является вспомогательной навигацией.

Правила:

- результаты поиска показывают полный путь категории;
- выбор результата раскрывает каскад до найденного элемента;
- найденная категория становится активной;
- пользователь не должен терять контекст после перехода из поиска.

## Продукты конечной категории

При выборе конечной категории интерфейс должен быть готов к отображению списка продуктов этой категории.

Список продуктов открывается рядом с каскадом категорий или в основной рабочей области справа.

Сведения о самой категории на рабочей странице каталога не являются приоритетными и не должны занимать основное место.

## Выравнивание каскадных колонок

При открытии дочернего уровня колонка не должна автоматически выравниваться по верхней границе предыдущей.

Следующий уровень открывается на вертикальном уровне выбранного элемента предыдущей колонки.

Это позволяет пользователю мгновенно видеть связь между выбранной категорией и её потомками без дополнительного поиска глазами.

Если высота экрана не позволяет полностью показать колонку, допускается корректировка положения таким образом, чтобы:

- выбранный элемент оставался связан с дочерней колонкой;
- колонка полностью помещалась в видимой области;
- связь между уровнями не терялась.

Допускается использовать тонкий визуальный коннектор (линия, точка, мягкая подсветка) между выбранным элементом и открывшейся колонкой, но он не должен становиться основным элементом интерфейса.

Основной способ показать связь — вертикальное выравнивание колонок.

## Переход от категорий к товарам

Каскадная навигация отображает только категории.

Товары не являются уровнем каскада и не должны открываться как дополнительная колонка внутри дерева категорий.

Правила:

- категория с дочерними категориями открывает следующий уровень каскада;
- категория без дочерних категорий считается конечным узлом;
- при выборе конечного узла пользователь переходит к списку товаров этой категории;
- список товаров отображается в разделе продуктов или в отдельной рабочей области, но не как часть иерархии категорий;
- заголовок «Товары» не должен появляться как колонка каскадной навигации.

## Индикаторы вложенности

Chevron справа показывает, что у категории есть дочерние категории.

Правила:

- chevron отображается на любом уровне вложенности;
- наличие chevron не зависит от номера колонки;
- если у категории нет дочерних категорий, chevron не отображается;
- точка, маркер или другой декоративный элемент не должен заменять chevron для категорий с дочерними элементами.

Логика отображения:

- есть дочерние категории → показываем chevron;
- нет дочерних категорий → chevron не показываем;
- конечный узел → переход к товарам по клику.
